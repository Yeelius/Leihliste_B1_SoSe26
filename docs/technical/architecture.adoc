== Architekturentscheidungen

_Architekturentscheidungen (Architecture Decision Records, ADRs) dokumentieren
wichtige technische Entscheidungen: Was wurde entschieden, warum, welche
Alternativen wurden verworfen, und welche Konsequenzen hat die Entscheidung._

_Jede Entscheidung, die schwer rückgängig zu machen ist oder das gesamte System
betrifft, sollte hier festgehalten werden. Kleine, lokale Implementierungsdetails
gehören nicht hierher._

NOTE: ADRs werden nicht gelöscht oder überschrieben — sie werden als
„ersetzt durch ADR-XXX" markiert. So bleibt die Entscheidungshistorie nachvollziehbar.

=== ADR-001: Verwendung von React für das Frontend

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 24.05.2026

| *Kontext*
| LeihListe benötigt eine moderne, gut bedienbare Weboberfläche (Frontend) für unterschiedliche Nutzungsszenarien. Nutzer sollen zentrale Funktionen wie Suche, Verfügbarkeitsprüfung, Ausleihe und Rückgabe effizient ausführen können. Daraus ergeben sich Anforderungen an Benutzbarkeit, mobile Nutzbarkeit, kurze Ladezeiten und eine wartbare UI-Struktur. Die Anwendung soll für verschiedene Nutzergruppen einsetzbar sein. Deshalb muss das Frontend flexibel genug sein, um unterschiedliche Rollen, Ansichten und Workflows abzubilden. Zusätzlich soll das Frontend eine spätere Umsetzung als Progressive Web App unterstützen.

| *Entscheidung*
| Wir verwenden React 19.2.x für das Frontend, weil React ein etabliertes Komponentenmodell bietet und die Entwicklung wiederverwendbarer, wartbarer und skalierbarer Benutzeroberflächen unterstützt. Dadurch können zentrale Elemente wie Inventarlisten, Such- und Filterfunktionen, Ausleihformulare, Statusanzeigen und rollenabhängige Ansichten strukturiert umgesetzt werden.

React ist weit verbreitet, besitzt eine große Community und bietet viele Vorlagen, Bibliotheken und Hilfsmittel für die Entwicklung moderner Webanwendungen. Außerdem lässt sich React gut mit responsiven Layouts und app-ähnlichen Weboberflächen kombinieren, wodurch die spätere mobile Nutzung von LeihListe unterstützt wird.

| *Alternativen*
| Eine Alternative wäre Vue.js. Vue bietet ebenfalls ein komponentenbasiertes Frontend-Modell und wäre für LeihListe grundsätzlich geeignet. Gegen Vue spricht, dass React weiter verbreitet ist, ein größeres Ökosystem besitzt und mehr Vorlagen, Bibliotheken und Hilfsmittel für moderne Webanwendungen bietet.

Eine weitere Alternative wäre eine serverseitig gerenderte Oberfläche mit Django Templates. Diese Lösung würde gut zu einem Python-Backend passen und wäre für einfache Verwaltungsseiten ausreichend. Gegen diese Alternative spricht jedoch, dass dynamische Oberflächen mit vielen Zuständen, Filtern, Verfügbarkeitsprüfungen und rollenabhängigen Ansichten mit React flexibler und übersichtlicher umgesetzt werden können.

| *Konsequenzen*
| Durch die Entscheidung für React wird die Entwicklung einer modularen und wiederverwendbaren Benutzeroberfläche erleichtert. Komponenten für Inventarlisten, Such- und Filterfunktionen, Ausleihformulare, Statusanzeigen und rollenabhängige Ansichten können mehrfach verwendet und einheitlich gestaltet werden.

Die Trennung zwischen React-Frontend und Python-Backend unterstützt eine klare Aufgabenverteilung im System. Das Frontend ist für die Darstellung und Benutzerinteraktion zuständig, während das Backend die Geschäftslogik, Datenhaltung und Schnittstellen bereitstellt.

Die Entscheidung erleichtert außerdem die spätere mobile Nutzung, da React gut mit responsiven Layouts und app-ähnlichen Weboberflächen kombiniert werden kann. Dadurch kann LeihListe perspektivisch auf Desktop-Geräten, Tablets und Smartphones genutzt werden.

Gleichzeitig entsteht zusätzlicher technischer Aufwand, da ein separates Frontend-Projekt mit Build-Prozess, Paketverwaltung und API-Kommunikation gepflegt werden muss. Außerdem müssen Abhängigkeiten regelmäßig aktualisiert werden, um Sicherheitsrisiken und Versionskonflikte zu vermeiden.
|===

=== ADR-002: Verwendung von Django für das Backend

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 24.05.2026

| *Kontext*
| LeihListe benötigt ein Backend zur Verwaltung von Nutzern, Rollen, Inventargegenständen, Ausleihen und Rückgaben. Dabei sind insbesondere Datenkonsistenz, Sicherheit, Wartbarkeit und eine klare Schnittstelle zum React-Frontend wichtig.

Da viele Funktionen wie Benutzerverwaltung, Rechteverwaltung, Datenbankzugriff und Verwaltungsoberflächen benötigt werden, soll ein Backend-Framework verwendet werden, das diese Grundfunktionen möglichst zuverlässig unterstützt.

| *Entscheidung*
| Wir verwenden Django für das Backend, weil Django viele zentrale Funktionen für Webanwendungen bereits mitbringt. Dazu gehören unter anderem Benutzerverwaltung, Rechte- und Rollenkonzepte, Datenbankanbindung über ein ORM (Object-Relational Mapping: Schicht zwischen dem Programmcode und der Datenbank) sowie Schutzmechanismen gegen typische Web-Schwachstellen.

Für die Kommunikation mit dem React-Frontend wird Django durch Django REST Framework ergänzt. Dadurch kann das Backend eine REST-API bereitstellen, über die das Frontend Inventardaten, Ausleihen, Reservierungen und Nutzerinformationen abrufen oder ändern kann.

| *Alternativen*
| Eine Alternative wäre FastAPI. FastAPI ist besonders für performante und schlanke APIs geeignet und bietet automatische API-Dokumentation. Gegen FastAPI spricht in diesem Projekt, dass viele benötigte Funktionen wie Benutzerverwaltung, Rollenmodell und Admin-Oberfläche stärker selbst zusammengestellt werden müssten als bei Django.

Eine weitere Alternative wäre Flask. Flask ist leichtgewichtig und flexibel, bringt aber weniger Funktionen standardmäßig mit. Für LeihListe würde dadurch mehr zusätzlicher Implementierungs- und Wartungsaufwand entstehen.

| *Konsequenzen*
| Durch die Entscheidung für Django können zentrale Backend-Funktionen schneller und strukturierter umgesetzt werden. Besonders Benutzerverwaltung, Rollen, Datenbankmodelle und administrative Funktionen profitieren von den vorhandenen Django-Komponenten.

Die Entscheidung unterstützt außerdem die Wartbarkeit, da Django eine klare Projektstruktur vorgibt und eine große Community besitzt. Probleme lassen sich dadurch häufig leichter recherchieren und lösen.

Gleichzeitig entsteht eine gewisse Abhängigkeit vom Django-Ökosystem. Außerdem muss die REST-Schnittstelle zum React-Frontend sauber entworfen werden, damit Frontend und Backend klar getrennt und langfristig wartbar bleiben.
|===

=== ADR-003: Datenbankschnittstelle via. MariaDB

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 24.05.2026

| *Kontext*
| Im Rahmen des Projekts Leihliste haben wir uns darauf festgelegt eine Datenbank als Speicher für verschiedene Daten zu verwenden. Dies ist relevant, da wir im Rahmen des Projekts zahlreiche Daten von Ausleihen, Nutzern und anderen Ressourcen speichern, bearbeiten, verarbeiten und strukturieren müssen. Diese sollte einfach einzurichten und gut dokumentiert sein mit niedriger Einstiegshürde für den Fall, dass man nicht auf bekannte Datenbanksysteme zurückgreift. Es handelt sich weiterhin um ein Projekt in kleinem Rahmen ohne große monitäre Unterstützung, weshalb wir versuchen die beste kostenlose Basis zu finden. Außerdem besitzen alle Beteiligten durch die Datenbanksysteme-Kurse Vorkenntnisse bezüglich SQL, obwohl es natürlich auch hier wieder verschiedene "SQL-Dialekte" gibt und man sich als Team einarbeiten muss in die jeweilig verwendete Datenbank. Die Verwendung einer Datenbank hilft hier insbesondere bei Nutzerrollen, denen man verschiedene Berechtigungen zuordnet, dem Mehrnutzerprinzip und der Datenkonsistenz.

| *Entscheidung*
| Wir verwenden MariaDB und damit einhergehend MySQL, weil dies eben beschriebene Anforderungen wie gute Dokumentation, Open-Source, Lizenzkostenfreiheit und auch für unser Projekt, welches eine Webanwendung werden soll, etabliert ist. Dementsprechend wird man zu auftauchenden Problemen in Nutzerforen genug Informationen finden um diese zu beheben. Des Weiteren [läuft diese gut auf der Hochschulumgebung (ungewiss)] und eignet sich auf für den Projektumfang hervorragend.

| *Alternativen*
| Mögliche Alternativen:
PostgreSQL ist eine gute und größer aufsetzbare Alternative, die aber komplizierter ist als MariaDB und vorerst der Einfachheit halber abgelehnt wurde.
SQLite ist eine sehr einfache Datenbank, die jedoch eingeschränkt skalierbar ist und auch die Mehrbenutzung limitiert.
MongoDB wäre eine nicht relationale Option gewesen, jedoch sind die Teamkenntnisse stark beschränkt auf diesem Bereich und dementsprechend hätte man potenzielle Einarbeitungsprobleme und höheren Zeitaufwand als ohnehin schon.
Microsoft SQL Server ist die Datenbanksoftware die wir bis dato auch in der Universität verwendet haben, jedoch ist die Standardversion kostenpflichtig (es sei denn man könnte dies über die Hochschule lösen).

| *Konsequenzen*
| Was wird einfacher?

Die Nutzung von MariaDB als verbreitetes Open-Source-Datenbanksystem sorgt für eine geringe Einstiegshürde dank existierender Teamkenntnisse im Bereich SQL und einer umfangreichen Community.
Das Lizenzkostenrisiko entfällt; die Lösung bleibt für das Projekt und zukünftige Erweiterungen kostenlos.
Gute Dokumentation und zahlreiche Online-Ressourcen ermöglichen schnelle Problemlösungen im Team.
Die Implementierung von Nutzerrollen, Mehrbenutzerbetrieb und Datenkonsistenz wird durch die relationale Struktur erleichtert.
Das System ist leicht in typische Hochschul- und Webanwendungs-Infrastrukturen integrierbar.
Was wird schwieriger oder teurer?

Installation, Betrieb und Wartung der Datenbank erfordern Grundkenntnisse in Administration, Datensicherung sowie Zugriffsschutz.
Die Unterschiede im SQL-Dialekt im Vergleich zu anderen Systemen (z. B. Microsoft SQL Server) können zu zusätzlichem Einarbeitungsaufwand und Fehlerquellen führen.
Bei Änderungen oder einem späteren Technologiewechsel kann ein Migrationsaufwand entstehen, da die Anwendung auf MariaDB ausgerichtet ist.
Neue Risiken:

Fehlerhafte Konfiguration oder unzureichende Absicherung der Datenbank bergen Risiken hinsichtlich Datensicherheit (z. B. Benutzerrechte, Zugriffsmanagement, Backups).
Abhängigkeit vom gewählten System: Bei spezifischen Erweiterungen oder bei wachsendem Projektumfang könnten spätere Limitationen auftreten, die einen Wechsel erschweren.

=== ADR-004: Docker

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 31.05.2026

| *Kontext*
| Das Projekt LeihListe besteht aus mehreren unterschiedlichen Komponenten: einem Frontend, einem Backend, einer Datenbank und einem Reverse-Proxy.
Da das Entwickler-Team unterschiedliche Betriebssysteme und lokale Setups nutzt, besteht die Gefahr von "Works on my machine"-Problemen, was die Entwicklung und das Onboarding erschwert.

Die manuelle Installation und Verknüpfung all dieser Laufzeitumgebungen und Abhängigkeiten für lokale Tests, automatisierte Tests (CI/CD) und das finale Deployment ist zu fehleranfällig und aufwendig.

| *Entscheidung*

Mit Docker Compose kann der gesamte Technologie-Stack (Frontend, Backend, Datenbank, Proxy) mit einem einzigen Befehl lokal gestartet und vernetzt werden, was die Entwicklung des Walking Skeletons und späterer Features massiv beschleunigt.

| *Alternativen*
| Eine Alternative wäre die **manuelle lokale Installation** aller Abhängigkeiten (Laufzeitumgebungen, Datenbankserver etc.) durch jeden Entwickler. Dies wurde verworfen, da es extrem fehleranfällig ist, zu Versionskonflikten im Team führt und das finale Deployment auf dem Hochschulserver unnötig verkompliziert.
Eine weitere Alternative wäre **Kubernetes (K8s)** zur Container-Orchestrierung. Dies wurde verworfen, da es für den Rahmen und die Skalierungsanforderungen dieses Hochschulprojekts völlig überdimensioniert ist und einen viel zu hohen administrativen Einarbeitungsaufwand erfordert.

| *Konsequenzen*
| *Positiv:* +
- Das "Works on my machine"-Problem wird eliminiert; der Code verhält sich auf jedem Rechner und Server exakt gleich. +
- Neue Entwickler können das gesamte Projekt innerhalb von Minuten aufsetzen, ohne Datenbanken oder Webserver manuell konfigurieren zu müssen. +
- Das spätere Deployment auf dem Hochschulserver wird stark vereinfacht, da nur die Docker-Umgebung als Voraussetzung gegeben sein muss. +
- Automatisierte Tests (Integrationstests) können in einer sauberen, jederzeit reproduzierbaren Container-Umgebung in GitHub Actions laufen.

*Negativ/Schwierig:* +
- Es entsteht initialer Aufwand, um `docker-compose.yml` (inkl. Netzwerke und Volumes) zu schreiben. +
- Das Team muss grundlegende Docker-Konzepte (Images, Container, Volumes) verstehen und bei Fehlern debuggen können.

*Risiken:* +
- Bei lokaler Entwicklung können Container viele Systemressourcen (RAM/CPU) binden. +
- Fehlerhafte Volume-Konfigurationen (z. B. bei der Datenbank) können lokal zu Datenverlust bei Neustarts führen. (Gegenmaßnahme: Saubere Dokumentation in der `deployment.adoc`).

[cols="1,3"]
|===
| *Status*

| *Datum*

| *Kontext*

Continuous Deployment (CD) GitHub Actions, weil sie in GitHub (der wichtigsten Arbeitsumgebung) direkt integriert sind, für öffentliche Repositories kostenlos genutzt werden können, gute Einarbeitbarkeit durch eine niedrige Einstiegshürde bieten und auch gut dokumentiert sind. Unerfahrene Teammitglieder könnten anfangs Schwierigkeiten bei der Fehleranalyse von fehlgeschlagenen Builds/Tests haben.
Bei privaten Repositories oder sehr vielen Build-Minuten könnten langfristig Kosten entstehen, aktuell betrifft das aber nicht unser Projekt.

*Neue Risiken:* +
Die Build-/Deployment-Prozesse werden zu einem zentralen Bestandteil des Workflows; falls die Pipeline Fehler enthält, kann dies die Entwicklung zeitweise blockieren.
Falsche Konfigurationen könnten versehentlich zu fehlerhaften Deployments oder fehlender Testabdeckung führen.
Bei hoher Nutzung können Limits der kostenlosen Nutzung erreicht werden

[cols="1,3"]
|===

| Akzeptiert

- Zentrale SSL/TLS-Verwaltung vereinfacht die Zertifikatsrotation +
- Statische Dateien werden schneller ausgeliefert (geringere Latenz für Nutzende) +
- Backend-Instanzen werden entlastet (Skalierbarkeit) +

*Negativ:* +
- Zusätzliche Komponente im Deployment erhöht die Komplexität des Setups +
- Fehlkonfigurationen können zu Sicherheitslücken führen (z. B. offene Ports) +
- Monitoring des nginx muss separat eingerichtet werden (z. B. Log-Analyse) -

*Risiken:* +
- Single Point of Failure: Bei Ausfall des nginx ist das gesamte System nicht erreichbar +
- Maßnahme: Hochverfügbarkeits-Setup mit mehreren nginx-Instanzen +
- Performance-Engpässe bei sehr hohem Traffic +
- Maßnahme: Lasttests in späterem Sprint durchführen und ggf. Caching einrichten / optimieren

=== ADR-012: DSGVO-konformes Datenmanagement durch Anonymisierung und Datensparsamkeit

[cols="1,3"]
|===
| *Status*
| Vorgeschlagen -> verschoben am Meeting am 14.07.2026

| *Datum*
| 07.06.2026

| *Kontext*
| Das System muss den Datenschutz-Anforderungen entsprechend der DSGVO entsprechen. Relevante Anforderungen für das Produkt sind: +
- Recht auf Löschung: Nutzer*innen müssen ihre Daten löschen lassen können. +
- Datensparsamkeit: Nur absolut notwendige personenbezogene Daten speichern. +
- Zweckbindung: Daten dürfen nur für den definierten Zweck (Ausleihverwaltung) genutzt werden. +

Daraus ergeben sich folgende Herausforderungen: +
- Die Ausleihhistorie muss für Verleihende erhalten bleiben (z.B. für Schadensfälle), darf aber keine personenbezogenen Daten enthalten. +
- Nutzerkonten sind für den Ausleihprozess essenziell, aber Felder wie z.B. Adresse sind nicht zwingend nötig. +
- Auch nach der Löschung von Nutzer*innen muss die Integrität der Datenbank gewahrt bleiben (z.B. Fremdschlüssel in der Ausleihe-Tabelle).

| *Entscheidung*
| Wir setzen Datensparsamkeit by Design um: + 
- Wir speichern nur zwingend notwendige Felder: Nutzer-ID, Mail, Name, Rolle +
- Wir vermeiden redundante Daten: Ausleihhistorie speichert nur Nutzer-ID +

Wir ersetzen bei Account-Löschung und Löschanfragen personenbezogene Daten (Anonymisierung), wenn diese aus anderen Gründen nicht mehr gespeichert werden müssen: +
- z.B. Name wird zu "Gelöschter Nutzer", andere Daten werden gelöscht +
- Nutzer-ID bleibt in der Ausleihhistorie erhalten, um die Integrität der Datenbank zu wahren +
- gelöschte Nutzer*innen können keine Aktivitäten mehr durchführen +
Wir definieren *Gründe, warum eine Löschung noch ausgesetzt wird*: +
- Die letzte Rückgabe ist noch nicht mehr als 30 Tage her, um Schadensmeldungen nachzuverfolgen.

| *Alternativen*
| *Hard-Deletes mit Datenbank-Backups:* Physische Löschung aller Nutzerdaten und die Ausleihhistorie wird in separaten Backups archiviert. Verworfen, weil verlagert Problem nur, denn Backups sind dann nicht DSGVO-konform. Außerdem würden Fremdschlüssel auf nicht-existente Nutzer*innen verweisen. +

*Soft-Deletes:* Markieren von Daten als "gelöscht" statt physischer Löschung. Verworfen, weil nicht DSGVO-konform, da die personenbezogenen Daten physisch erhalten bleiben.

*Pseudonymisierung statt Anonymisierung:* Personenbezogene Daten werden durch reversible Pseudonyme ersetzt. Verworfen, weil Re-Identifizierung bleibt möglich, sodass auch nicht DSGVO-konform.

| *Konsequenzen*
| *Positiv:* +
- DSGVO-Konformität: Recht auf Löschung wird umgesetzt, ohne die Ausleihhistorie zu zerstören +
- Datenintegrität: Fremdschlüssel bleiben erhalten, Referenzen in der Ausleihhistorie bleiben valide +
- Wartbarkeit: Keine manuellen Backups oder Archivierungen nötig +

*Negativ:* +
- Umgang mit Lösch-Anfragen, die zeitlich noch gegen Speicher-Gründe widersprechen: Nutzer*innen müssen warten, bevor sie den Account löschen können bzw. Lösch-Anfragen müssen von Betreiber*innen aufbewahrt und später umgesetzt werden

*Risiken:* +
- Re-Identifizierung anonymisierter Daten, also nicht nur Nutzer-ID hochzählen, sodass rekonstruiert werden kann, wer Nutzer 1 war: *Nutzer-IDs dürfen nicht fortlaufend, aber müssen einmalig sein*

=== ADR-013: Speicherung von Medien und Dokumenten über die Datenbank statt Cloud-Speicher

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 07.06.2026

| *Kontext*
a| Für Ausleihobjekte können neben Stammdaten auch Bilder oder Dokumente benötigt werden, z. B. Fotos eines Gegenstands, Zustandsdokumentationen oder einfache Nachweise.

Ursprünglich wurde in ADR-008 eine getrennte Datenspeicherung mit Cloud-Speicher für Mediendaten vorgeschlagen. Dieses Vorgehen wurde verworfen, weil es zusätzliche Komplexität, Datenschutzfragen, mögliche Kosten und Synchronisationsaufwand zwischen Datenbank und Cloud-Speicher erzeugt.

Für die aktuelle Skalierung des Produkts ist kein externer Cloud-Speicher notwendig. Die Anwendung soll zunächst auf Hochschulinfrastruktur betrieben werden und möglichst einfach, nachvollziehbar und wartbar bleiben.

| *Entscheidung*
a| Wir speichern Medien und Dokumente nicht in einem externen Cloud-Speicher, sondern verwalten sie über die bestehende Datenbank bzw. das Datenbankmodell.

Damit bleibt die Datenhaltung für die aktuelle Projektgröße einfach und zentral. Bilder, Dokumente und zugehörige Metadaten können gemeinsam mit den fachlichen Daten des Ausleihobjekts verwaltet, gesichert und gelöscht werden.

Wir verwenden diese Lösung, weil sie:

- zur aktuellen Skalierung des Produkts passt
- keine zusätzliche Cloud-Infrastruktur benötigt
- besser zu On-Premise-Hosting auf Hochschulinfrastruktur passt
- die technische Komplexität reduziert
- die Nachvollziehbarkeit von Datenhaltung, Backups und Löschung vereinfacht

| *Alternativen*
a| - *Externer Cloud-Speicher für Mediendaten:* Diese Alternative wurde verworfen, weil zusätzliche Kosten, Datenschutzprüfung, Anbieterabhängigkeit und Synchronisation zwischen Datenbank und Cloud-Speicher entstehen würden.
- *Getrennte Speicherung personenbezogener Daten in der Datenbank und Mediendaten in der Cloud:* Diese Alternative entspricht dem abgelehnten ADR-008. Sie wurde verworfen, weil zwei Speichersysteme konsistent gehalten werden müssten und dadurch mehr Fehlerquellen entstehen.
- *Dateisystem auf dem Server ohne klare Datenbankanbindung:* Diese Alternative wurde verworfen, weil Dateien und Datenbankeinträge leichter auseinanderlaufen können, wenn Löschung, Backup oder Migration nicht sauber koordiniert werden.

| *Konsequenzen*
a| *Positiv:*

- keine Abhängigkeit von einem externen Cloud-Anbieter
- geringerer Implementierungs- und Wartungsaufwand
- einfachere DSGVO-Prüfung, weil Daten auf der vorgesehenen Infrastruktur bleiben
- einfachere lokale Entwicklung und Tests mit Docker Compose
- Medien und fachliche Daten können gemeinsam verwaltet werden

*Negativ/Schwierig:*

- große Dateien können die Datenbank belasten
- Ladezeiten können schlechter werden, wenn Bilder nicht optimiert oder zu groß gespeichert werden
- Speicherbedarf der Datenbank wächst mit der Anzahl der Medien

*Risiken:*

- Bei starkem Wachstum kann die Datenbank durch viele oder große Mediendateien zu groß werden
 * Gegenmaßnahme: Dateigrößen begrenzen, Bildkomprimierung nutzen und bei späterem Wachstum eine neue ADR für objektbasierten Speicher prüfen.
- Backups können größer und langsamer werden
 *  Gegenmaßnahme: Backup-Strategie früh mit dem Betrieb auf Hochschulinfrastruktur abstimmen.

=== ADR-014: Client-Server Webanwendung als grundlegender Architekturstil

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 07.06.2026

| *Kontext*
a| Unser Projekt soll eine Plattform zur Verwaltung und Nachverfolgung 
von Ausleihen für Hochschulen und private Nutzergruppen bereitstellen.
Bei der grundlegenden Architekturentscheidung stellt sich die Frage, ob das 
System als native Desktop-Software, als mobile App oder als Client-Server 
Webanwendung realisiert werden soll.

Die Anforderungen aus dem Produktkonzept sind eindeutig:

- Betrieb auf Hochschulinfrastruktur (kein App-Store, keine lokale Installation)
- Plattformunabhängigkeit (Windows, macOS, Linux, Android, iOS)
- Mehrere Nutzerrollen (Admin, Verleiher, Ausleiher) mit zentraler Datenhaltung
- Mobile Nutzung für schnelle Ausleihe vor Ort (z. B. mit Smartphone)
- Zentrale Inventarverwaltung, die für alle Nutzer gleichzeitig aktuell ist

Eine lokale Desktop-Anwendung kann diese Anforderungen strukturell nicht 
erfüllen, da sie keine zentrale Datenhaltung und keinen plattformübergreifenden 
Zugriff ermöglicht.

| *Entscheidung*
a| Wir bauen LeihListe als **Client-Server Webanwendung**, weil sie:

- **Plattformunabhängig** ist: jedes Gerät mit Browser kann das System 
  nutzen ohne Installation
- **Zentrale Datenhaltung** ermöglicht: alle Nutzer sehen denselben 
  aktuellen Inventarstand
- **Rollenbasierter Zugriff** einfach umsetzbar ist: Authentifizierung 
  und Autorisierung erfolgen serverseitig
- **Auf Hochschulinfrastruktur** betreibbar ist: nur ein Server nötig, 
  kein App-Store, keine Verteilung
- **Mobile Nutzung** Browser auf Smartphone reicht aus (PWA)
- **Wartung zentralisiert** ist: Updates werden einmal auf dem Server 
  eingespielt, alle Nutzer profitieren sofort

| *Alternativen*
a| Eine Alternative wäre eine **native Desktop-Anwendung** (z.B. mit Java 
Swing oder Electron). Dies wurde verworfen, weil:

- Jeder Nutzer die Software manuell installieren und aktualisieren müsste
- Keine zentrale Datenhaltung ohne zusätzlichen Server möglich ist
- Plattformspezifische Builds für Windows, macOS und Linux nötig wären
- Mobile Nutzung vor Ort nicht möglich wäre

Eine weitere Alternative wäre **App-Store-Applikation oder Play-Store-Applikation**
 (z.B. mit Kotlin für Android oder Swift für Apple/macOS)
Dies wurde verworfen, weil:

- Separate Entwicklung für Android und iOS den Projektrahmen sprengt
- Veröffentlichung im App-Store mit hohen Hürden und Kosten verbunden ist
- Das Team keine nativen App-Entwicklungskenntnisse (Swift, Kotlin) besitzt

| *Konsequenzen*
a| *Positiv:*

- Jeder Nutzer kann das System sofort über den Browser nutzen, keine 
  Installation, kein Update-Aufwand
- Zentrale Datenhaltung garantiert konsistenten Inventarstand für alle 
  Nutzer gleichzeitig, Race Conditions werden serverseitig behandelt
- Ein einziger Deployment-Prozess auf dem Hochschulserver reicht für 
  alle Plattformen
- PWA ermöglicht Installation auf dem Homescreen ohne App-Store
- Rollenbasierter Zugriff (Admin, Verleiher, Ausleiher) ist serverseitig  sauber trennbar

*Negativ/Schwierig:* +
- Internetverbindung ist Voraussetzung, Offline-Nutzung nur eingeschränkt möglich (Gegenmaßnahme: PWA-Caching für Kernfunktionen, offline-Ansicht, siehe ADR-011) +
- Serverseitige Sicherheit muss sorgfältig implementiert werden: 
  Eingabevalidierung, Authentifizierung, HTTPS zwingend erforderlich +
- Seitenladezeit < 2s muss aktiv optimiert werden

*Risiken:* +
- Bei Serverausfall ist das System für alle Nutzer gleichzeitig nicht   erreichbar (Gegenmaßnahme: Verfügbarkeitsziel ≥ 99% durch stabile   Hochschulinfrastruktur, siehe Produktkonzept) +
- Browser-Kompatibilität muss aktiv getestet werden, insbesondere   Safari auf iOS (Gegenmaßnahme: Cross-Browser-Tests ab Sprint 3,   siehe ADR-011)

=== ADR-015: Monolithische Backend-Struktur statt Microservices

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 07.06.2026

| *Kontext*
| Das Backend von unserem Projekt muss mehrere fachliche Domänen abdecken:
Nutzerverwaltung, Inventarverwaltung, Ausleihprozess, Benachrichtigungen 
und QR-Code Generierung.

Bei der Strukturentscheidung des Backends stellt sich die Frage, ob diese  Domänen als einzelne unabhängige Microservices oder als ein zusammenhängender  Monolith realisiert werden sollen.

Die Rahmenbedingungen des Projekts sind:
- Hochschulprojekt mit begrenzter Projektlaufzeit (6 Sprints über zwei Semester)
- Entwicklerteam 7 Personen ohne Microservices-Erfahrung
- Betrieb auf Hochschulinfrastruktur mit begrenzten Serverressourcen
- Überschaubare Nutzerzahl (Hochschule + private Nutzergruppen)

Eine Microservices-Architektur würde bedeuten, dass z.B. 
Nutzerverwaltung, Ausleihservice und Benachrichtigungsservice als 
separate deploybare Einheiten mit eigener Datenbank und eigenem 
Containern betrieben werden müssten.

| *Entscheidung*
| Wir verwenden eine **monolithische Backend-Struktur** mit Django, 
weil sie: +
- **Entwicklungskomplexität reduziert** — ein Codebase, ein Deployment, 
  eine Datenbank statt verteilte Systeme +
- **Zum Projektrahmen passt** — Hochschulprojekt mit begrenzter Zeit und  Ressourcen sowie kleine Teamgröße mit bescheidener Programmiererfahrung würde das höchstwahrscheinlich nicht schaffen +
- **Intern modular strukturiert** ist — fachliche Domänen werden als 
  Module (Dateien) getrennt, ohne verteilte Komplexität +
- **Auf Hochschulinfrastruktur** problemlos betreibbar ist — ein 
  einziger Backend-Container reicht +
- **Datenkonsistenz vereinfacht** — alle Domänen teilen eine Datenbank, 
  keine verteilten Transaktionen nötig +
- **Spätere Migration ermöglicht** — bei Bedarf können einzelne Module 
  später zu Microservices extrahiert werden

| *Alternativen*
| Eine Alternative wäre eine **Microservices-Architektur** mit separaten 
Services pro Domäne (z. B. Auth-Service, Inventory-Service, 
Notification-Service). Dies wurde verworfen, weil: +
- Der operative Aufwand (Service Discovery, API Gateway, verteiltes   Logging, Inter-Service-Kommunikation) den Projektrahmen bei weitem 
  überschreitet +
- Das Team keine Erfahrung mit verteilten Systemen hat —   Einarbeitungsaufwand zu hoch +
- Datenkonsistenz zwischen Services (verteilte Transaktionen,   Eventual Consistency) für ein Ausleihsystem kritisch und komplex ist +
- Für uns zur Verfügung gestellter Server könnte diese Architektur einfach viel zu schwer sein, sodass auch das Programm beim Benutzen nicht flüssig ist.

| *Konsequenzen*
| *Positiv:* +
- Ein einziger `python manage.py runserver` Befehl startet das gesamte 
  Backend — kein verteiltes Setup nötig +
*Hinweis*: `runserver` wird für die lokale Ausführung, für die Entwicklung und im nächsten Semester auch für das Testen gebraucht. Beim Deployment auf dem HTW-Server und bei der Inbetriebnahme der Webanwendung für Nutzer wird `runserver` nicht gebraucht, weil die Anwendung mit Docker Compose in Containern ausgeführt wird. +
- Datenbankzugriffe über Django ORM sind direkt und konsistent — keine verteilten Transaktionen +
- Neue Teammitglieder verstehen die gesamte Backend-Logik in einem  Codebase ohne verteilte Systemkenntnisse +
- Deployment auf Hochschulserver ist ein einziger Container (siehe ADR-004 Docker) +
- Fachliche Trennung bleibt erhalten durch Django Apps als Module:  'users', 'Inventar', 'Ausleihe', 'Benachrichtigungen', 'QR-Code'

*Negativ/Schwierig:* +
- Bei starkem Wachstum der Nutzerzahl kann das gesamte Backend zum   Flaschenhals werden — einzelne Domänen können nicht unabhängig   skaliert werden +
- Fehler in einer Domäne können potenziell das gesamte Backend   beeinflussen — sorgfältige Fehlerbehandlung nötig +
- Codebase wächst mit der Zeit — Modulgrenzen müssen diszipliniert   eingehalten werden

*Risiken:* +
- Unklare Modulgrenzen können später zu Problemen führen —   unkontrolliertes Verwachsen der Domänen; als Gegenmaßnahme: Klare   Django App Struktur von Beginn an +
- Datenbankschema wächst mit allen Domänen gemeinsam —   Migrationen müssen sorgfältig koordiniert werden; Gegenmaßnahme: Jede Django App verwaltet eigene Migrationen
|===

=== ADR-018: Datenkonsistenz und Vermeidung von Race Conditions beim Ausleihprozess

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 07.06.2026

| *Kontext*
| unsere LeihListe verwaltet einen zentralen Inventarbestand, auf den mehrere  Nutzer gleichzeitig zugreifen können. Eine kritische Situation entsteht  wenn zwei oder mehr Nutzer denselben Gegenstand zeitgleich ausleihen wollen.

*Beispielszenario:*

Zeitpunkt 1: Nutzer A prüft → Laptop-1 ist verfügbar 
Zeitpunkt 1: Nutzer B prüft → Laptop-1 ist verfügbar 
Zeitpunkt 2: Nutzer A leiht aus → Laptop-1 wird reserviert
Zeitpunkt 2: Nutzer B leiht aus → Laptop-1 wird nochmal reserviert 
Ergebnis: Laptop-1 ist doppelt verliehen — Datenkonsistenz verletzt

Dieses Problem wird als **Race Condition** bezeichnet — zwei parallele 
Prozesse lesen denselben Zustand und schreiben ihn gleichzeitig zurück 
ohne voneinander zu wissen.

Die Anforderungen aus dem Produktkonzept sind: +
- Datenkonsistenz — jeder Gegenstand hat einen eindeutigen Status in  der Datenbank, der für alle Nutzer verbindlich gilt +
- Keine Race Conditions — das System muss technisch ausschließen, dass derselbe Gegenstand doppelt verliehen wird +
- Nachvollziehbarkeit — jede Statusänderung wird mit Zeitstempel und  Nutzerreferenz gespeichert, um Schadensfälle lückenlos klären zu können

| *Entscheidung*
| Wir verhindern Race Conditions durch eine Kombination aus drei Maßnahmen:

**1. Optimistic Locking** +
Jeder Gegenstand trägt eine Versionsnummer in der Datenbank. 
Beim Ausleihvorgang wird geprüft ob die Version zum Zeitpunkt 
des Speicherns noch mit der Version übereinstimmt die der Nutzer 
beim Öffnen gesehen hat: +
- Stimmt die Version überein → Ausleihe wird durchgeführt, 
  Versionsnummer wird erhöht +
- Stimmt die Version nicht mehr überein → ein anderer Nutzer 
  hat den Gegenstand zwischenzeitlich ausgeliehen → 
  der Nutzer erhält eine Fehlermeldung und kann es erneut versuchen

**2. Atomare Datenbankoperationen** +
Django sperrt die betroffene Datenbankzeile für die Dauer der Transaktion — kein anderer Prozess kann sie gleichzeitig ändern. 
Erst nach Abschluss der Transaktion wird die Sperre freigegeben.

**3. Serverseitige Statusverwaltung** +
Jeder Gegenstand durchläuft einen klar definierten Statusfluss in der Datenbank: verfügbar → ausgeliehen → zurückgegeben

Statusänderungen dürfen ausschließlich durch kontrollierte Backend-Logik ausgelöst werden. Das Frontend zeigt den aktuellen Status nur an — eine direkte Manipulation des Status durch den Client ist technisch nicht möglich. So wird sichergestellt dass kein Statuswechsel an der Geschäftslogik vorbei stattfinden kann.

Wir verwenden diese Kombination weil sie: +
- Datenkonsistenz auf Datenbankebene garantiert — unabhängig  von der Anzahl gleichzeitiger Nutzer +
- Mit Django ORM direkt umsetzbar ist — keine zusätzlichen   Bibliotheken nötig +
- Performance nicht stark beeinträchtigt — Sperren bestehen nur für  kurze Transaktionsdauer +
- Nachvollziehbarkeit ermöglicht — jeder Statuswechsel wird mit Zeitstempel und Nutzer-ID in der Historie gespeichert

| *Alternativen*
| Eine Alternative wäre **kein Locking** — einfach zuerst speichern gewinnt. Dies wurde verworfen, weil:
- Race Conditions nicht verhindert werden — doppelte Ausleihe möglich, der Ausleiher, der nicht ausleihen konnte, ist dann frustriert +
- Das Vertrauen der Nutzer in das System leiden würde +
- Datenkonsistenz nicht garantiert ist

Eine weitere Alternative wäre **Pessimistic Locking** — der Gegenstand 
wird bereits beim Anzeigen gesperrt. Dies wurde verworfen, weil:
- Nutzer A schaut sich Laptop-1 an → Laptop-1 ist für alle anderen 
  gesperrt — schlechte User Experience

Eine weitere Alternative wäre eine **Redis-basierte Sperrverwaltung** 
(Distributed Lock). Dies wurde verworfen, weil: +
- Ein zusätzlicher Redis-Server nötig wäre — erhöht Deployment-Komplexität +
- Die Lösung für die erwartete Nutzerzahl überdimensioniert ist +
- Djangos eingebaute Mechanismen für unsere Skalierung ausreichen

Eine weitere Alternative wäre eine **Queue-basierte Verarbeitung** — 
Ausleiheanfragen werden nacheinander in einer Warteschlange abgearbeitet. 
Dies wurde verworfen, weil: +
- Zusätzliche Message-Queue Infrastruktur (z. B. Celery Bibliothek was Redis braucht, was wiederum einen eigenen Server braucht) nötig wäre +
- Die Latenz für Nutzer steigt — Anfragen müssen in der Queue warten +
- Der Ansatz für ein Hochschulprojekt überdimensioniert ist

| *Konsequenzen*
| *Positiv:* +
- Race Conditions auf Datenbankebene vollständig verhindert —  doppelte Ausleihe desselben Gegenstands ist technisch unmöglich +
- Lückenlose Historie aller Statuswechsel — Transparenz in  Schadensfällen gewährleistet (Anforderung aus Produktkonzept) +
- Direkt mit Django ORM umsetzbar — kein zusätzlicher Infrastrukturaufwand +
- Nutzer bekommt sofortiges Feedback wenn ein Gegenstand bereits   vergeben ist

*Negativ/Schwierig:* +
- Atomare Sperren funktionieren nur mit Datenbanken die Row-Level 
  Locking unterstützen — MariaDB unterstützt dies (siehe ADR-003) +
- Bei sehr hoher gleichzeitiger Last können kurze Wartezeiten entstehen 
  während eine Zeile gesperrt ist +
- Optimistic Locking erfordert saubere Fehlerbehandlung im Frontend — 
  Nutzer muss informiert werden wenn die Ausleihe fehlschlägt

*Risiken:*
- Deadlocks können entstehen wenn eine Transaktion mehrere Sperren 
  gleichzeitig hält und zwei Transaktionen gegenseitig aufeinander 
  warten — keine kann weitermachen +
  (Gegenmaßnahme: Transaktionen so kurz und fokussiert wie möglich 
  halten, nie mehr als eine Sperre pro Transaktion) +
- Erhält ein Nutzer eine Fehlermeldung, ist die natürliche Reaktion 
  mehrfach zu klicken — das sendet mehrere identische Anfragen ans 
  Backend und erhöht die Last unnötig +
  (Gegenmaßnahme: Button nach dem ersten Klick sofort deaktivieren 
  bis das Backend geantwortet hat — Umsetzung durch das Frontend-Team)
